## 测试指标体系与论文图表设计方案

本方案整合了之前的讨论，围绕三组对比实验（无治理、静态限流、动态定价），为你提供一套清晰、完整的测试指标体系以及论文可用的图表建议。

---

### 一、测试目标

- 验证动态定价治理（Rajomon）相比无治理和静态限流，在系统保护、延迟稳定性、公平性等方面的优势。
- 量化动态定价对突发负载的响应能力（价格变化、拒绝率、恢复时间）。
- 展示动态定价在真实负载模式下的鲁棒性和可观测性。

---

### 二、实验设计

#### 2.1 对比组设置

| 组名                 | 描述                                     | 目的                                   |
| -------------------- | ---------------------------------------- | -------------------------------------- |
| **基线组（无治理）** | MCP 服务器不加任何拦截器                 | 获取系统原始性能上限和过载崩溃行为     |
| **静态限流组**       | MCP 服务器 + 拦截器启用固定 QPS 阈值限流 | 与工业常用方案对比，突出动态定价的优劣 |
| **动态定价组**       | MCP 服务器 + 拦截器启用 Rajomon 动态定价 | 验证核心治理方案                       |

#### 2.2 负载模式

| 模式         | 描述                                                       | 考察点                     |
| ------------ | ---------------------------------------------------------- | -------------------------- |
| **阶梯负载** | 逐步增加并发（如 10→30→80→150 请求/秒），每阶段持续 2 分钟 | 系统吞吐量拐点、过载行为   |
| **突发负载** | 短时间（如 30 秒）流量激增至峰值的 10 倍                   | 瞬时过载保护、价格响应速度 |
| **泊松负载** | 请求到达间隔服从泊松分布，平均率固定（如 50 请求/秒）      | 随机波动下的稳定性、公平性 |

#### 2.3 运行次数

每组每种负载模式运行 3 次，取平均值，确保可复现性。

---

### 三、测试指标体系

#### 3.1 治理效果指标（核心）

| 指标类别     | 具体指标           | 定义/计算方法                          | 用途                         |
| ------------ | ------------------ | -------------------------------------- | ---------------------------- |
| **性能**     | 吞吐量（QPS）      | 单位时间成功请求数                     | 对比三组在高负载下的处理能力 |
|              | 延迟分布           | 平均延迟、P50/P95/P99 延迟             | 动态定价是否能稳定延迟       |
|              | 错误率             | 5xx 响应比例                           | 是否避免系统崩溃             |
| **治理效果** | 拒绝率             | 429 响应比例                           | 保护强度对比                 |
|              | 动态价格变化       | 实时价格（从响应头或 Prometheus 获取） | 验证定价算法对负载的响应     |
|              | 公平性             | 不同预算组（如 10/50/100）的成功率     | 动态定价是否实现分级保护     |
|              | 优先级调度（可选） | 不同优先级任务的成功率                 | 若实现优先级，验证调度效果   |
| **稳定性**   | 恢复时间           | 负载回落后，延迟/价格恢复正常所需时间  | 系统弹性                     |
|              | 资源利用率         | CPU、内存使用率                        | 治理是否避免资源耗尽         |

#### 3.2 拦截器开销指标（可选，用于说明）

| 指标           | 定义                                     | 测量方法                                     |
| -------------- | ---------------------------------------- | -------------------------------------------- |
| 拦截器处理延迟 | 拦截器自身代码执行耗时                   | 代码级埋点（如记录进入和离开拦截器的时间差） |
| 内存占用       | 拦截器维护的滑动窗口、价格表等占用的内存 | 运行时内存分析工具                           |
| CPU 开销       | 拦截器占用的 CPU 比例                    | 与无治理组对比资源使用差异                   |

### 四、数据收集

#### 4.1 每请求日志（CSV 字段）

| 字段                           | 说明                        |
| ------------------------------ | --------------------------- |
| `timestamp`                    | 请求结束时间（毫秒）        |
| `client_budget`                | 请求携带的 Token 预算       |
| `latency_ms`                   | 请求耗时                    |
| `status_code`                  | HTTP 状态码                 |
| `price_returned`               | 响应头中的 Price 值（若有） |
| `token_consumed`               | 响应头中的 X-Token-Usage 值 |
| `rejected`                     | 是否被拒绝（status=429）    |
| `task_id` / `step_idx`（可选） | Agent 场景时记录任务信息    |

#### 4.2 Prometheus 指标（网关暴露）

| 指标名                             | 类型      | 标签                                 |
| ---------------------------------- | --------- | ------------------------------------ |
| `rajomon_requests_total`           | Counter   | status（accepted/rejected）、handler |
| `rajomon_request_duration_seconds` | Histogram | handler                              |
| `rajomon_token_usage`              | Histogram | handler                              |
| `rajomon_current_price`            | Gauge     | 无                                   |
| `rajomon_composite_cost`           | Gauge     | 无                                   |

#### 4.3 资源监控

- 使用 `docker stats` 或 `cadvisor` 收集容器级 CPU、内存使用率，时间粒度 1 秒。

### 五、预期结果对比

| 对比维度       | 无治理                 | 静态限流                             | 动态定价                                           |
| -------------- | ---------------------- | ------------------------------------ | -------------------------------------------------- |
| **低负载**     | 正常，低延迟           | 正常，低延迟                         | 正常，价格低，无拒绝                               |
| **突发负载**   | 延迟飙升，可能崩溃     | 超过阈值后所有请求被拒（吞吐量跌零） | 价格快速上涨，低预算请求被拒，高预算成功，延迟平稳 |
| **高负载持续** | 系统不可用（高错误率） | 吞吐量为 0，系统空闲                 | 分级拦截，系统稳定，高预算成功率明显高于低预算     |
| **负载回落**   | 延迟缓慢下降           | 恢复后重新允许请求                   | 价格快速下降，请求逐渐恢复正常                     |
| **公平性**     | 所有用户同等受损       | 无区分，阈值后全部拒绝               | 高预算 > 中预算 > 低预算                           |
| **资源利用率** | CPU 飙升可能耗尽       | CPU 受控但可能空闲                   | CPU 平稳，利用率高                                 |

### 六、论文可出的图表

#### 6.1 吞吐量与拒绝率时间序列图

- **内容**：x 轴为时间，左 y 轴为成功/拒绝请求数（堆叠面积或两条线），右 y 轴为动态价格（仅动态定价组）。
- **用途**：直观展示三组在突发负载下的行为差异（无治理崩溃、静态限流断崖、动态定价平滑）。
- **图例**：三条曲线分别代表三种策略，或分三个子图。

#### 6.2 延迟累计分布图（CDF）

- **内容**：x 轴为延迟（ms），y 轴为累积概率，绘制三组策略的 P50/P95/P99 延迟。
- **用途**：对比延迟分布，证明动态定价能有效控制高负载下的长尾延迟。

#### 6.3 公平性箱线图

- **内容**：x 轴为预算组（10/50/100），y 轴为成功率，绘制静态限流和动态定价两组（无治理可略）。
- **用途**：验证动态定价是否实现分级保护，高预算用户成功率更高。

#### 6.4 动态价格响应曲线

- **内容**：x 轴为时间，左 y 轴为请求率，右 y 轴为价格，绘制动态定价组的价格随请求率变化曲线。
- **用途**：展示定价算法对负载的实时响应速度和灵敏度。

#### 6.5 系统恢复时间对比（可选）

- **内容**：柱状图，x 轴为策略，y 轴为负载回落后延迟/价格恢复到正常水平所需时间。
- **用途**：评估系统的弹性。

#### 6.6 资源利用率曲线

- **内容**：x 轴为时间，y 轴为 CPU 使用率（%），叠加三种策略的曲线。
- **用途**：验证治理是否避免资源耗尽，且动态定价下资源利用率更平稳。

#### 6.7 参数敏感性分析（可选，针对动态定价）

- **内容**：子图展示不同 α 值（平滑因子）对价格响应速度、系统稳定性的影响。
- **用途**：体现对算法的深入分析，提升论文深度。

### 七、实施建议

1. **先完成基础实验**：用阶梯负载和突发负载快速验证三组对比，获取核心图表。
2. **后扩展 Agent 场景**：如需提升论文创新性，再实现 Agent 模拟器，增加任务级指标和图表。
3. **数据后处理**：使用 Python（pandas, matplotlib, seaborn）批量生成图表，保持风格一致。
4. **论文写作**：实验部分按照“实验设置 → 指标定义 → 结果分析 → 讨论”结构组织，每个图表配以文字解读。
